# Goal

Implement a local-PC POC of a reasoned trajectory program for openpilot master.

Scope for this phase:

- Run only on this PC, not on C3X and not through an eGPU path.
- Keep panda, opendbc, CAN safety, and car interface command composition untouched.
- Insert upstream of `controlsd` by publishing a bounded `lateralManeuverPlan`.
- Generate or receive a compact `RTPv1` trajectory program, parse it with a strict grammar, and reject invalid or late output.
- Compile RTP fields into deterministic candidate path constraints and costs over the openpilot base plan.
- Publish same-frame, valid, deadline-met plans by default. For local-PC POC VLMs that cannot meet 50 ms synchronously, allow an explicit bounded async mode with `rtpSourceFrameId` / `rtpAgeFrames` freshness accounting and fail-closed stale output.
- Log RTP fields, selected candidate, latency, deadline misses, invalid RTP count, path delta, and speed delta in `reasonedTrajectoryPlan`.

The POC uses a pluggable local VLM backend. With `RTP_VLM_COMMAND` set, the external engine receives a fixed scene-board payload and is pinned to the local PC GPU by default with `CUDA_VISIBLE_DEVICES=0`. Without that command, the local demo uses a static RTP program so the parser, compiler, barrier, and publisher path can be verified without a downloaded VLM.

Current measured VLM status on this PC:

- Qwen2.5-VL-3B via Transformers is too slow for 20 Hz synchronous use. It can run only as a bounded async backend in this POC. The sim loop remains nonblocking because the planner consumes the latest bounded RTP while tracking `rtpSourceFrameId` / `rtpAgeFrames`.
- Direct freeform Qwen RTP generation is still not reliable: the BASE example biases greedy decoding toward nominal output, while removing it can produce malformed or out-of-bounds RTP. The fixed POC therefore uses `qwen_label_rtp_worker.py`: Qwen emits bounded scene labels from the UI-overlay scene board, then deterministic code compiles those labels into strict valid `RTPv1`. The default VLM run path is now full384 rotating score mode: `--image-mode full --label-mode score --score-rotate-groups`, with all score labels durable through short occlusion/splash by default. These are now worker defaults; `--label-mode generate`, `--no-score-rotate-groups`, `--image-mode composite`, and `--image-mode multi` remain opt-in experiments.

- Root cause diagnosis remains useful: PathSynth/control was not the blocker. The blocker was Qwen perception/decoding and input framing. The current best default is a full driver-UI-style overlay board at 384 px, constrained label scoring/generation, and deterministic grammar-bounded RTP compilation. Composite/crop modes remain available as opt-in experiments, not defaults. This is intentionally a POC path, not a final 20 Hz ONNX/TensorRT implementation.
- SmolVLM2-500M via Transformers is too slow for 20 Hz and did not emit valid RTP in the local benchmark.
- SmolVLM2-256M ONNX Runtime CUDA runs but is still hundreds of milliseconds per frame. Lower scene-board resolution helps CPU/render/input overhead but the exported vision encoder still takes fixed 512x512 tensors.
- SmolVLM2-256M ONNX Runtime TensorRT with q4f16 fails because TensorRT cannot import ORT contrib `MatMulNBits` / `MultiHeadAttention` nodes. The fp16 decoder crashes the Windows TensorRT EP during session creation. A hybrid TensorRT-vision/CUDA-decoder path runs, but still misses 50 ms.
- nanoVLM-230M PyTorch single-crop is the fastest tested candidate, but even the 0-token/1-token lower-bound harness is not a valid multi-field RTP generator at the required 50 ms p99. It remains unsuitable for the control-influence demo unless paired with a separate bounded detector/compiler path.

The sim harness now uses actual MetaDrive camera frames with a driver-UI-style model overlay, plus a route-following MetaDrive controller. Static RTP verifies nonzero path influence in sim; Qwen verifies a real VLM backend in the bounded async loop, but not same-frame 20 Hz reasoning.
